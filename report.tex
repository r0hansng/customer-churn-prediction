\section{Problem Statement}
% ═══════════════════════════════════════════════════════════════════════════════

Customer churn --- the voluntary discontinuation of service by a subscriber --- is one of the most economically damaging phenomena in the telecommunications industry. Acquiring a new customer is estimated to cost five to seven times more than retaining an existing one, and industry research suggests that even a \textbf{5\% reduction in churn can increase profitability by 25--95\%}.

\subsection{Challenge Definition}

Given a rich set of 19 customer attributes spanning demographics, service subscriptions, account details, and billing behaviour, the challenge is to build a reliable binary classification system that answers:

\begin{quote}
\itshape
\textbf{``Will this customer churn within the next billing cycle?''}
\end{quote}

More specifically, the system must:
\begin{enumerate}[leftmargin=2em]
    \item \textbf{Predict} whether a given customer will churn (\texttt{Churn = Yes}) or remain (\texttt{Churn = No}).
    \item \textbf{Quantify} the probability of churn to enable risk-tiered retention prioritisation.
    \item \textbf{Operate} in two modes: real-time single-customer inference and bulk batch processing.
    \item \textbf{Generalise} robustly to unseen customer profiles without overfitting to training noise.
\end{enumerate}

\subsection{Business Impact}

Accurate churn prediction allows the business team to:
\begin{itemize}[leftmargin=2em]
    \item Target high-risk customers with personalised retention offers \emph{before} they leave.
    \item Prioritise customer success resources efficiently.
    \item Identify systemic issues driving disengagement (e.g., contract type, internet service quality).
\end{itemize}

\subsection{Class Imbalance Handling}

The dataset exhibits moderate class imbalance with a 73.5\% / 26.5\% split. Three complementary strategies were applied:
\begin{enumerate}[leftmargin=2em]
    \item \textbf{\texttt{class\_weight=`balanced'}} on applicable models (Logistic Regression, Decision Tree, Random Forest): automatically adjusts per-class loss penalties inversely proportional to class frequencies.
    \item \textbf{SMOTE} (Synthetic Minority Oversampling Technique) via \texttt{imblearn.Pipeline}: generates synthetic churn examples inside each CV training fold only, preventing test-set contamination.
    \item \textbf{ROC-AUC} as the primary CV scoring metric: unlike accuracy or F1, ROC-AUC is threshold-agnostic and correctly handles class imbalance.
\end{enumerate}

\section{Methodology}
% ═══════════════════════════════════════════════════════════════════════════════

\subsection{System Architecture}

The system is structured as a two-component pipeline:

\begin{enumerate}[leftmargin=2em]
    \item \textbf{Offline Training} (\texttt{src/preprocessing/preprocess.py} + \texttt{src/evaluation/train.py}): Builds and serialises scikit-learn \texttt{Pipeline} objects to \texttt{.joblib} files.
    \item \textbf{Online Inference} (\texttt{app.py}): Loads \texttt{.joblib} files and serves predictions via a Streamlit web application with two modes: single-customer and batch CSV.
\end{enumerate}

\subsection{Preprocessing Pipeline}

To ensure a \textbf{leak-proof} preprocessing strategy (i.e., fitting scalers only on training data), all transformations are encapsulated inside a \texttt{ColumnTransformer} that becomes part of each model's \texttt{Pipeline}:

\begin{lstlisting}[language=Python, caption={Preprocessing Pipeline (src/preprocessing/preprocess.py)}]
numeric_transformer = Pipeline([
    ('imputer', SimpleImputer(strategy='median')),
    ('scaler',  StandardScaler())
])

categorical_transformer = Pipeline([
    ('imputer', SimpleImputer(strategy='most_frequent')),
    ('onehot',  OneHotEncoder(handle_unknown='ignore'))
])

preprocessor = ColumnTransformer([
    ('num', numeric_transformer, numeric_features),    # tenure, MonthlyCharges, TotalCharges
    ('cat', categorical_transformer, categorical_features)   # 16 categorical columns
])
\end{lstlisting}

\subsubsection{Design Decisions}
\begin{itemize}[leftmargin=2em]
    \item \textbf{Median imputation} for \texttt{TotalCharges}: robust to the skewed distribution caused by early-churning customers.
    \item \textbf{StandardScaler} for numerical features: makes regularised models (LR, MLP) converge faster and improves coefficient interpretability.
    \item \textbf{One-Hot Encoding with \texttt{handle\_unknown='ignore'}}: prevents errors when serving predictions on new customers with unseen category values.
    \item \textbf{Train/test split}: 80\% / 20\% stratified by random seed 42 (5,634 train / 1,409 test).
\end{itemize}

\subsection{Feature Engineering}

Five domain-driven features were derived from the raw columns and added to the numeric pipeline:

\begin{table}[H]
\centering

\subsection{Classifier Selection \& Rationale}

\begin{table}[H]
\centering
\caption{Model Selection Rationale}
\begin{tabularx}{\textwidth}{llX}
    \toprule
    \textbf{Model} & \textbf{Library Class} & \textbf{Rationale} \\[4pt]
    \midrule
    Logistic Regression
        & \texttt{LogisticRegression}

        & \texttt{DecisionTreeClassifier}
        & Captures non-linear interactions; produces human-readable rules. Pruning via \texttt{max\_depth} prevents overfitting. \\[6pt]

All classifiers are wrapped in an \texttt{imblearn.Pipeline}:
\[
\texttt{ImbPipeline} = [\; \underbrace{\texttt{ColumnTransformer}}_{\text{Preprocessing}} \;\rightarrow\; \underbrace{\texttt{SMOTE}}_{\text{Oversampling}} \;\rightarrow\; \underbrace{\texttt{Classifier}}_{\text{Model}} \;]
\]
SMOTE fires only during \texttt{.fit()} inside each CV fold and is bypassed at inference time, preventing data leakage.

\subsection{Metrics Used}

Given the class imbalance (26.5\% positive class), multiple complementary metrics were used:

\begin{itemize}[leftmargin=2em]
    \item \textbf{ROC-AUC}: Primary CV optimisation target. Threshold-agnostic; measures ranking ability across all thresholds. Unaffected by class imbalance.
    \item \textbf{Recall}: Of all actual churners, how many were caught? The most business-critical metric.
    \item \textbf{F1 Score}: Harmonic mean of precision and recall; reported for comparison.
    \item \textbf{Accuracy}: Overall fraction correct. Reported for completeness but not relied upon due to imbalance.
\end{itemize}

\section{Optimisation}
% ═══════════════════════════════════════════════════════════════════════════════

\subsection{Hyperparameter Tuning Strategy}

All five models were optimised using \textbf{RandomizedSearchCV with 5-fold cross-validation}, scoring on ROC-AUC to correctly handle the imbalanced nature of the dataset. For algorithms lacking native \texttt{class\_weight} support, SMOTE inside an imblearn \texttt{Pipeline} ensures proper minority-class attention:

\begin{lstlisting}[language=Python, caption={RandomizedSearchCV Setup (src/evaluation/train.py)}]
search = RandomizedSearchCV(
    estimator=pipeline,                # Preprocessor -> SMOTE -> Classifier
    param_distributions=config['params'],
    scoring='roc_auc',                 # Optimise for ROC-AUC
    cv=5,                              # 5-fold CV
    n_iter=config['n_iter'],           # Random trials per model
    n_jobs=-1,
    random_state=42
)
search.fit(X_train, y_train)
best_pipeline = search.best_estimator_
\end{lstlisting}

\subsection{Optimisation Impact \& Design Rationale}

\begin{itemize}[leftmargin=2em]
    \item \textbf{Logistic Regression}: Achieved the best CV ROC-AUC overall. Setting \texttt{class\_weight=balanced} was sufficient to handle imbalance natively.
    \item \textbf{Decision Tree \& Ensembles}: Constraining tree depth (\texttt{max\_depth=5} for DT, \texttt{10} for RF, \texttt{3} for GB) mitigated overfitting and improved validation scores. Random Forest and Gradient Boosting yielded the highest accuracy on the test set.
    \item \textbf{MLP}: The smaller \texttt{(50,)} hidden layer converged better, while using SMOTE correctly handled the class imbalance.
    \item \textbf{Cross-validation}: 5-fold CV using ROC-AUC ensures selection of models that separate the classes effectively regardless of threshold. Subsequent tuning of the threshold to 0.35 significantly improves recall for real-world application.
\end{itemize}

\section{Milestone 2: Agentic AI Retention Engine}
% ═══════════════════════════════════════════════════════════════════════════════

\subsection{Overview}

Milestone 2 extends the prediction system into an \textbf{agentic AI retention strategist}. When a high-risk customer is flagged ($P(\text{churn}) > 0.50$), a LangGraph workflow retrieves the most relevant retention policies from a FAISS document store and generates a personalised, policy-grounded strategy using Google Gemini 2.5 Flash.

\subsection{System Architecture}

\begin{enumerate}[leftmargin=2em]
    \item \textbf{RAG Retrieval Node} (\texttt{src/retention/vector\_store.py}): Converts the customer profile into a natural-language query, embeds it with Gemini Embedding-001, and performs a FAISS cosine similarity search against \texttt{telecom\_retention\_policies.md} to retrieve the top-2 relevant policy chunks.
    \item \textbf{Strategy Generation Node} (\texttt{src/retention/graph\_engine.py}): Sends the customer profile, churn probability, and retrieved policy context to Gemini 2.5 Flash. The model returns a bullet-pointed, policy-cited, personalised retention plan.
\end{enumerate}

The two nodes are connected in a \texttt{StateGraph} (LangGraph), compiled into a callable agent:
\[
\text{START} \;\rightarrow\; \underbrace{\texttt{retrieve\_policies}}_\text{RAG node} \;\rightarrow\; \underbrace{\texttt{generate\_strategy}}_\text{LLM node} \;\rightarrow\; \text{END}
\]

\subsection{Streamlit Integration}

The retention engine is surfaced in two modes within the Streamlit dashboard:

\begin{enumerate}[leftmargin=2em]
    \item \textbf{Single Prediction} (\texttt{ui/single\_prediction.py}): After a high-risk prediction is confirmed by the ensemble, a ``Generate Retention Strategy (AI)'' button appears. Clicking it invokes \texttt{run\_retention\_engine()} and renders the Gemini-generated strategy inline with Markdown formatting.
    \item \textbf{Batch Prediction} (\texttt{ui/batch\_prediction.py}): After running predictions on an uploaded CSV, a ``Generate Collective Retention Strategy (AI)'' button appears for any batch containing predicted churners (Section~\ref{sec:batch-retention}).
\end{enumerate}

API keys are retrieved securely from Streamlit Secrets on Streamlit Cloud, with a local \texttt{.env} fallback for development.

\subsection{Batch Collective Retention Strategy}
\label{sec:batch-retention}

For bulk inference use cases it is impractical and expensive to generate an individual Gemini API call per churner --- a dataset with 7,043 customers and 26.5\% churn rate would require $\approx 1{,}866$ sequential LLM requests. Instead, the system follows an \textbf{aggregate-then-generate} pattern:

\begin{enumerate}[leftmargin=2em]
    \item \textbf{Segment Profiling} (\texttt{graph\_engine.\_build\_segment\_profile()}): After batch predictions complete, all rows predicted to churn are isolated. Fifteen aggregate statistics are computed across the at-risk population:
    \begin{itemize}[leftmargin=2em]
        \item Numerical: average churn probability, average tenure, average monthly charges, average active service count.
        \item Categorical (mode): dominant contract type, internet service tier, payment method.
        \item Proportional: \% Month-to-month, \% Fiber optic, \% Electronic check, \% Senior Citizen, \% No Partner, \% Paperless Billing.
    \end{itemize}
    \item \textbf{RAG Retrieval} (\texttt{\_retrieve\_batch\_policies()}): The segment profile is serialised to a natural-language string and used to retrieve the top-3 most relevant policy chunks from the FAISS index --- one extra chunk compared to the single-customer case to capture broader policy coverage.
    \item \textbf{Strategy Generation} (\texttt{\_generate\_batch\_strategy()}): A single \texttt{Gemini 2.5 Flash} call receives the aggregate profile and retrieved policies. The prompt instructs the model to produce a structured retention programme covering:
\end{enumerate}

\[
\text{START} \;\rightarrow\; \underbrace{\texttt{\_retrieve\_batch\_policies}}_{\text{RAG node}} \;\rightarrow\; \underbrace{\texttt{\_generate\_batch\_strategy}}_{\text{LLM node}} \;\rightarrow\; \text{END}
\]

\begin{tcolorbox}[colback=lightgray, colframe=primary, title={\textbf{Batch Strategy Output Structure}}]
\begin{enumerate}[leftmargin=1.5em, itemsep=2pt]
    \item \textbf{Root Cause Analysis} --- why this segment is leaving.
    \item \textbf{Recommended Retention Initiatives} --- exact offers, discounts, contract upgrades, citing specific company policies.
    \item \textbf{Estimated Impact} --- projected retention lift if initiatives are executed.
    \item \textbf{Sub-Segment Targeting} --- differentiated treatment for demographic sub-groups (e.g.\ senior citizens, high-bill customers, no-partner customers).
\end{enumerate}
\end{tcolorbox}

\textbf{Cost efficiency}: regardless of batch size, exactly \textbf{one Gemini API call} is made. The approach shifts the analytical burden from the LLM to a deterministic aggregation function, preserving statistical representativeness while eliminating per-record API overhead.

\section*{Conclusion}
\addcontentsline{toc}{section}{Conclusion}
% ═══════════════════════════════════════════════════════════════════════════════

This report documents the complete end-to-end development of the Customer Churn Prediction \& Retention Strategy system across both milestones. Key outcomes:

\begin{itemize}[leftmargin=2em]
    \item A \textbf{leak-proof imblearn Pipeline} combining feature engineering, SMOTE, and \texttt{class\_weight} handling ensures correct treatment of class imbalance.
    \item \textbf{Five classifiers} tuned via RandomizedSearchCV (ROC-AUC): Logistic Regression (best Recall: 78.88\%, 89.57\% at threshold 0.35), Random Forest (best F1: 62.05\%), and Gradient Boosting (best CV ROC-AUC: 84.59\%).
    \item Churn-class \textbf{Recall improved by +13--23 percentage points} over the Milestone 1 baseline through SMOTE, class weighting, and feature engineering.
    \item A \textbf{LangGraph + RAG retention engine} (Gemini 2.5 Flash + FAISS) generates personalised, policy-grounded retention plans for individually flagged high-risk customers.
    \item A \textbf{Batch Collective Retention Strategy} aggregates all predicted churners into a single segment profile and generates one cost-effective, population-level retention programme via a single Gemini API call --- efficient regardless of batch size.
    \item A \textbf{fully integrated Streamlit dashboard} provides single-customer inference with ensemble verdict and AI strategy, bulk batch processing with collective retention programme, and a metrics comparison page.
    \item The system is \textbf{deployment-ready} on Streamlit Community Cloud with secure API key handling via Streamlit Secrets.
\end{itemize}
